# Полусеместровый контроль №4 — поиск аномальных респондентов SoS

Проект решает задачу поиска аномально высокой активности респондентов в поисковых запросах SoS.
Алгоритм находит случаи, когда один респондент в конкретный день слишком сильно влияет на OTS отдельного бренда внутри категории.

## Что делает решение

1. Загружает исходные данные из папки `data_train/` или архива `data_train.zip`.
2. Оставляет строки, которые входят в поставку SoS: `BrandinDelivery = 1`.
3. Считает дневной OTS на уровне:

```text
SubjectID + researchdate + BrandID + CategoryDelivery
```

4. Находит аномально высокие значения `daily_ots` с помощью статистического порога внутри каждой категории.
5. Записывает в `anomalies.csv` пары `SubjectID + researchdate`, потому что удаление применяется ко всем активностям респондента за этот день.
6. Сохраняет подробное объяснение каждой аномалии в `anomaly_reasons.csv`.
7. Строит три обязательных графика до/после очистки.

## Запуск

Если рядом с решением лежит папка `data_train/`:

```bash
python solution.py
```

Если данные лежат в архиве:

```bash
python solution.py --data data_train.zip
```

После запуска создаётся папка `output/` с результатами.

## Почему решение валидно

Алгоритм не использует вручную заданные `SubjectID`, даты, бренды или фиксированный процент удалений.
Порог рассчитывается из данных, отдельно для каждой категории.
Малые значения OTS не попадают в аномалии, потому что удаляются только наблюдения, у которых `daily_ots` превышает верхний статистический порог.
Для каждой найденной аномалии сохраняются `daily_ots`, `score`, `threshold` и текстовая причина удаления.

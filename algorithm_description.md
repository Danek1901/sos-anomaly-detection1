# Описание алгоритма поиска аномальных респондентов

## Цель

Нужно найти респондентов, которые в конкретный день создают аномально высокий вклад в OTS отдельного бренда внутри категории.
Такие респонденты могут искажать отчетную выборку, поэтому они удаляются из анализа целиком за соответствующий день.

## Единицы анализа

Аномалия ищется на уровне:

```text
SubjectID + researchdate + BrandID + CategoryDelivery
```

Итоговая единица удаления:

```text
SubjectID + researchdate
```

Если респондент признан аномальным хотя бы по одному бренду в один день, в итоговый файл `anomalies.csv` попадает только пара `SubjectID + researchdate`.
При последующей очистке удаляются все строки этого респондента за этот день, включая другие бренды и категории.

## Подготовка данных

В анализ берутся только строки, которые соответствуют условиям:

```text
BrandinDelivery = 1
CategoryDelivery не пустая
```

Если в файле категория называется `CategoryNameDelivery`, код приводит её к рабочему виду и использует как аналог `CategoryDelivery`.

## Расчет daily_ots

Для каждой группы считается количество строк исходного датасета:

```text
count_rows = число запросов респондента по бренду и категории за день
```

Дневной OTS считается по формуле:

```text
daily_ots = Weight * count_rows
```

Пример: если вес респондента равен 5, а по бренду за день найдено 3 строки, то `daily_ots = 15`.

## Score и порог

Для каждой категории отдельно строится распределение значений `daily_ots`.
Так как OTS имеет тяжелый хвост, перед расчетом порога используется логарифм:

```text
log_ots = log1p(daily_ots)
```

Порог внутри категории:

```text
threshold_log = max(Q99.9, Q3 + 2 * IQR)
```

где:

```text
Q99.9 — 99.9-й перцентиль;
Q1 — первый квартиль;
Q3 — третий квартиль;
IQR = Q3 - Q1.
```

После этого порог переводится обратно в шкалу OTS:

```text
threshold = exp(threshold_log) - 1
```

Итоговый score:

```text
score = daily_ots / threshold
```

Аномалия фиксируется, если:

```text
score > 1
count_rows >= 2
```

Первое условие означает, что дневной OTS респондента выше верхнего статистического порога.
Второе условие защищает от ситуации, когда респондент удаляется из-за одного случайного запроса с большим весом.

## Почему малый OTS не попадает в аномалии

Алгоритм ищет только превышение верхнего порога.
Если `daily_ots` маленький, то `score = daily_ots / threshold` будет меньше 1, поэтому такая строка не попадет в `anomaly_reasons.csv`.
Дополнительное условие `count_rows >= 2` также не дает удалять респондентов из-за единичной слабой активности.

## Почему подход интерпретируемый

Для каждой аномалии сохраняются:

```text
SubjectID
researchdate
BrandID
Brand
CategoryDelivery
daily_ots
score
threshold
reason
```

По этим полям можно объяснить, какой бренд стал причиной удаления, насколько большим был OTS и какой порог был превышен.

## Почему нет подгонки

В решении нет вручную заданных ID, дат, брендов, категорий или фиксированного процента удалений.
Количество найденных аномалий получается только из данных и выбранного статистического правила.
Алгоритм детерминированный: при повторном запуске на тех же данных результат будет одинаковым.
